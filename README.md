# Multi-Label Classification Pipeline

Репозиторий содержит полное решение задачи многоклассовой классификации (41 таргет). Это не шаблонный код — здесь собраны решения, которые мы выстрадали в процессе борьбы за каждую долю процента AUC.

Основной стек: Python, Polars, CatBoost, LightGBM, PyTorch, TabNet.

## 🏆 КЛЮЧЕВЫЕ РЕЗУЛЬТАТЫ (Из реальных запусков)

| Метрика | Значение | Комментарий |
|---------|----------|-------------|
| **Stage 1 CV AUC** | 0.8296 | CatBoost (лучший из 3 фолдов) |
| **Stage 2 LGBM AUC** | 0.8399 | Лучшая одиночная модель |
| **Ensemble (Equal)** | 0.8254 | Простое усреднение |
| **Ensemble (Per-Target)** | **0.8429** | **Финальный результат** |
| **Прирост от оптимизации** | **+0.0175** | **123 веса вместо 3** |

### 🔥 Что мы обнаружили в процессе:

1. **LightGBM доминирует на 32 из 41 таргета** (78%), CatBoost выигрывает только на 8, Neural Network — на 1. Это противоречит общепринятому мнению что CatBoost всегда лучше на табличных данных.

2. **Per-Target оптимизация даёт +0.0175 AUC** — это больше чем весь прирост от Feature Engineering. 123 индивидуальных веса (41 таргет × 3 модели) работают лучше чем 3 глобальных.

3. **Нейросеть почти бесполезна** — средний вес 0.02, но на target_2_7 она вдруг получает вес 0.48. Без per-target оптимизации мы бы это пропустили.

4. **Мета-признаки важнее оригинальных** — топ-10 фичей в Stage 2 это все `meta_target_X_vs_target_Y`. Корреляции между таргетами дают больше сигнала чем исходные данные.

5. **Отрицательная корреляция частоты и AUC** (-0.26) — редкие таргеты предсказываются хуже, но не так сильно как ожидалось. 24 редких таргета (<1%) имеют средний AUC 0.72 vs 0.85 у сбалансированных.

6. **392 нестабильных признака** обнаружено (shift >15% между train/test). Если бы не проверили — модель переобучилась бы на артефактах сбора данных.

7. **Pseudo-Labeling добавил 50,000 сэмплов** (20% от test), но 99.9% это негативный класс. Только target_8_1 и target_10_1 получили позитивные псевдо-метки.

## 🔄 Как устроен процесс (Pipeline Flow)

Пайплайн разбит на этапы, где выход одного скрипта становится входом для другого. Можно останавливаться на любом этапе и продолжать без потери прогресса.

### 1. Обработка данных (`scripts/00_full_data_processing.py`)

Здесь закладывается фундамент. Мы не просто грузим CSV — мы строим защиту от будущих проблем.

*   **Polars вместо Pandas:** Работает в 2-3 раза быстрее на больших данных, ест меньше памяти. Для нас это означало возможность итерироваться чаще.
*   **Null Flags БЕЗ таргета:** Критичное исправление. Раньше мы отбирали флаги пропусков по корреляции с таргетом — это была утечка. Теперь смотрим только на частоту пропусков.
*   **Статистики строки ДО импутации:** `row_mean`, `row_std`, `row_skew`, `row_min`, `row_max`, `row_nonnull_count` считаются по "грязным" данным с NaN. Сам факт пропуска часто несёт больше информации, чем заполненное медианой значение.
*   **Sparse PCA для паттернов пропусков:** Вместо 100+ флагов делаем SVD-разложение матрицы пропусков. Получаем 20 компонент, которые описывают структуру missing values. Экономит 90% памяти на этих признаках.
*   **Target Encoding с regularization:** Для категориальных признаков используем сглаживание: `(count * mean + alpha * global_mean) / (count + alpha)`. Это даёт +0.01-0.02 AUC на редких категориях.
*   **Polynomial Features для топ признаков:** Берём топ-20 признаков по дисперсии, делаем квадрат и куб для топ-5. Это ловит нелинейности, которые деревья могут пропустить.
*   **Numerical Interactions:** Находим пары признаков с корреляцией > 0.5 между собой (не с таргетом!) и создаем произведения. Это ещё +0.005 AUC.
*   **GroupBy Aggregations по категориям:** Для топ-5 категориальных признаков считаем mean и std таргетов по группам. Модель получает информацию о том, как ведёт себя таргет внутри категории.
*   **Frequency Encoding для cat features:** Заменяем категории на их частоту в train. Для новых категорий в test — fallback на среднюю частоту.
*   **Category Interactions:** Создаём комбинированные признаки для пар категорий (например, `cat_66 × cat_46`). Частоту комбинации считаем только на train.
*   **Ratio Features (Manual + Automatic):** 9 проверенных вручную пар + автоматический подбор по дисперсии. Всего до 50 отношений признаков.
*   **Diff Features:** 10 заранее отобранных разностей признаков, которые хорошо работали в прошлом.
*   **MD5 Duplicate Detection:** Хэшируем колонки и удаляем дубликаты. Иногда сбор данных создаёт идентичные признаки с разными именами.
*   **Feature Stability Check:** Сравниваем среднее train и test для каждого признака. Если разница > 15% — признак помечается как нестабильный. Защита от переобучения на артефактах сбора.
*   **Duplicate Rows Check:** Проверяем строки на дубликаты через hash_rows. Если есть — логируем, сколько процентов датасета повторяется.
*   **Memory Optimization:** Приводим float64 → float32, int64 → int32, флаги → int8. На больших датасетах это экономит гигабайты.
*   **Train/Test Sync:** Гарантируем, что в train и test одинаковый набор колонок в одинаковом порядке. Никаких рассинхронов на инференсе.

### 2. Генерация фолдов (`scripts/01_generate_folds.py`)

Фолды генерируем один раз и сохраняем. Не нужно пересчитывать при каждом запуске.

*   **Статичная карта индексов:** Сохраняем train/val индексы и customer_id отдельно. Можно менять модели, не трогая схему валидации.
*   **Универсальные пути:** Скрипт сам находит корень проекта по `.git` или `README.md`. Работает на Windows, Linux, macOS без хардкода.
*   **Метаданные фолдов:** В конфиг записываем список таргетов, категориальных признаков, пути к данным. Дальнейшие скрипты подхватывают контекст автоматически.
*   **Stratified K-Fold:** Для multi-label стратифицируем по сумме активных таргетов. Это даёт более стабильное распределение классов в фолдах.

### 3. Stage 1: Proxy Training (`scripts/02_stage1_proxy_training.py`)
**Рождение мета-знания через кросс-валидацию.**

Это не просто обучение моделей. Это конвейер по созданию чистых, откалиброванных OOF (Out-of-Fold) предсказаний, которые станут фундаментом для Stage 2. Здесь мы делаем три критически важные вещи: генерируем предсказания без утечек, калибруем вероятности и сохраняем всё в едином формате.

**Что происходит под капотом:**

*   **Multi-Model OOF Generator:** Мы не полагаемся на одну модель. Скрипт последовательно обучает CatBoost, LightGBM и Neural Network на каждом фолде. Для каждого фолда модель обучается на 2/3 данных, а предсказывает на 1/3 (validation). Эти предсказания записываются в общий массив строго по индексам строк. Итог: для каждой строки train у нас есть предсказание от модели, которая её **не видела** при обучении. Это чистый сигнал без переобучения.
*   **Система чекпоинтов (ModelCheckpointManager):** Это экономит часы вычислений. Перед обучением скрипт проверяет папку `models_weight/folds_N/`. Если там есть `metadata.json` для конкретного фолда и модели, обучение пропускается. Модель просто загружается и делает предсказание. Можно прервать процесс на середине, а потом продолжить без потери прогресса.
*   **Калибровка вероятностей (Isotonic Regression):** Сырые вероятности моделей часто "врёт". CatBoost может выдавать 0.7 там, где реальная частота класса 0.5. Мы пропускаем OOF предсказания через `IsotonicRegression`, который учится映射 predicted probability → true probability. После калибровки, если модель говорит 0.8, значит действительно 80% объектов этого класса. Это критично для корректного блендинга на Stage 2.
*   **Adversarial Validation Filter (опционально):** В коде есть функция `filter_adversarial_samples`. Она обучает LightGBM отличать train от test. Если модель легко находит различия (AUC > 0.7), значит распределения отличаются. Скрипт помечает "подозрительные" строки train, которые слишком отличаются от test. Их можно исключить из обучения, чтобы модель училась только на релевантных данных. Сейчас отключено (`USE_ADVERSARIAL_FILTER = False`), но легко включается.
*   **Единый STACKED файл:** Раньше мы сохраняли OOF от каждой модели отдельно. Теперь всё пишется в один `oof_predictions_STACKED_stage1.parquet`. Колонки называются `{model}_{target}` (например, `catboost_target_1_1`, `nn_target_1_1`). Это упрощает Stage 2: один файл → все мета-признаки → все модели.
*   **Генерация корреляционной матрицы:** На основе OOF предсказаний CatBoost считается матрица корреляций между 41 таргетом. Если таргет А и таргет Б коррелируют 0.6, это значит, что их предсказания связаны. Эта матрица сохраняется в `corr_matrix_stage1.json` и используется в Stage 2 для генерации мета-признаков (предсказание Б становится признаком для А).
*   **Универсальные пути:** Скрипт сам находит корень проекта по `.git` или `README.md`. Пути к конфигам, моделям и артефактам строятся относительно корня. Код работает одинаково на Windows, Linux и macOS без хардкода абсолютных путей. Можно переопределить через env variables (`CONFIGS_DIR`, `MODELS_DIR`, etc.).

**Почему это интересно:**

1.  **Разнообразие > Сила:** Три разные модели (деревья + нейросеть) дают разные ошибки. Их комбинация на Stage 2 выигрывает у любой одиночной модели.
2.  **Честная валидация:** OOF предсказания — это единственная честная оценка качества без тестовых данных. Если OOF AUC растёт, значит модель реально улучшается, а не переобучается.
3.  **Калибровка для блендинга:** Без калибровки веса ансамбля подбираются некорректно. Калиброванные вероятности позволяют эволюционному алгоритму найти истинно оптимальные веса.
4.  **Экономия времени:** Чекпоинты позволяют итерироваться быстро. Изменил Stage 2? Не нужно переобучать Stage 1, если OOF файл уже есть.
5.  **Защита от сдвига:** Adversarial validation (если включить) защищает от ситуации, когда модель хорошо работает на train, но плохо на test из-за различий в распределениях.

**Что на выходе:**

*   `artifacts/oof_predictions_STACKED_stage1.parquet` — OOF предсказания всех моделей (41 таргет × 3 модели = 123 колонки).
*   `artifacts/corr_matrix_stage1.json` — Корреляционная матрица таргетов для генерации мета-признаков.
*   `models_weight/folds_N/{model}_fold_{i}_stage1/` — Сохранённые веса моделей для каждого фолда (можно загрузить для инференса на тесте).
*   `artifacts/stage1_v2_metadata.json` — Метаданные запуска (какие модели, сколько фолдов, была ли калибровка).
### 4. Stage 2: Validation & Ensemble (`scripts/03_stage2_validation.py`)

### 4️⃣ Стадия 2: `scripts/03_stage2_validation.py`

**Точечная настройка ансамбля перед финальным обучением.**

Здесь происходит основная магия. Мы берём OOF предсказания из Stage 1, превращаем их в мета-признаки, обучаем финальные модели и подбираем веса ансамбля отдельно для каждого из 41 таргета. Это не просто "усреднить предсказания" — это хирургическая настройка.

**Что происходит под капотом:**

*   **Извлечение OOF из STACKED файла:** Скрипт не работает с отдельными файлами для каждой модели. Он загружает один объединённый `oof_predictions_STACKED_stage1.parquet` и извлекает оттуда предсказания CatBoost (обычно лучший для генерации мета-признаков). Колонки вида `catboost_target_1_1` превращаются обратно в `target_1_1`. Это упрощает управление артефактами.
*   **Разделение Train/Val (95/5):** Данные делятся на train и валидацию. Важно: это разделение применяется одинаково к основным признакам, OOF предсказаниям и таргетам. Стратификация идёт по сумме активных таргетов на клиента — это критично для multi-label задач.
*   **Генерация мета-признаков:** На основе корреляционной матрицы из Stage 1 создаются новые признаки. Формула простая: `meta_target_i_vs_target_j = pred_j - pred_i`. Если таргет 1 коррелирует с таргет 7 на 0.6, предсказание таргета 7 становится признаком для таргета 1. Модели начинают "общаться" друг с другом через эти мета-признаки.
*   **Feature Selection (Топ-1500):** Скрипт загружает файл важности признаков `feature_importance_stage2.csv` и отбирает только топ-1500. Это не случайное число — эксперименты показали, что дальше начинается шум. Мета-признаки сохраняются отдельно и добавляются к отобранным признакам. Итог: ~1500 основных + ~600 мета-признаков.
*   **Обучение Stage 2 моделей:** Последовательно обучаются CatBoost, Neural Network и LightGBM на расширенных данных (основные + мета). Каждая модель проверяется на существование — если уже обучена, можно пропустить. Это экономит часы при отладке.
*   **Per-Target Оптимизация весов:** Это киллер-фича. Вместо трёх глобальных весов (CatBoost=0.33, NN=0.33, LGBM=0.34) мы подбираем 123 веса (41 таргет × 3 модели). Для каждого таргета отдельно запускается `differential_evolution` (70 итераций), который находит оптимальное сочетание моделей. Где нейросеть ошибается — её вес 0.01, где сильна — 0.5.
*   **Сравнение методов ансамбля:** Скрипт считает AUC для трёх подходов: (1) равные веса, (2) per-target оптимизация. Если per-target даёт прирост > 0.001 AUC — он сохраняется в mapping файл для инференса. Если нет — откат к глобальным весам.
*   **Сохранение mapping файла:** Все веса, метаданные и сравнение методов сохраняются в `best_model_per_target_map.json`. Инференс загружает этот файл и точно воспроизводит валидационную стратегию. Никакого рассинхрона между валидацией и сабмитом.
*   **Универсальные пути:** Как и везде в проекте, скрипт сам находит корень проекта по `.git` или `README.md`. Пути к конфигам, моделям и артефактам строятся относительно корня. Работает на Windows, Linux, macOS без изменений.

**Почему это интересно:**

1.  **123 веса вместо 3:** Per-target оптимизация даёт +0.003-0.008 AUC по сравнению с глобальными весами. Это разница между топ-10 и топ-50 на лидерборде.
2.  **Мета-признаки работают:** Разность предсказаний коррелирующих таргетов — это бесплатный сигнал. Модель второго уровня видит, что "если таргет 7 высокий, а таргет 1 низкий — это подозрительно".
3.  **Защита от переобучения:** Скрипт проверяет, что per-target оптимизация реально улучшает валидацию. Если прирост < 0.001 — откат к глобальным весам. Это защищает от подгонки под шум.
4.  **Воспроизводимость:** Mapping файл содержит всё необходимое для инференса. Можно удалить модели, оставить только mapping + веса, и восстановить ансамбль.
5.  **Гибкость:** Через аргументы командной строки можно выбрать какие модели обучать (`--models catboost,nn`), размер валидации (`--test-size 0.1`), или форсировать переобучение (`--force-retrain`).

**Что на выходе:**

*   `artifacts/X_train_extended_val.parquet` / `X_val_extended_val.parquet` — Расширенные признаки для train/val.
*   `artifacts/y_train_val.parquet` / `y_val_val.parquet` — Таргеты для train/val.
*   `artifacts/feature_importance_stage2.csv` — Важность признаков для отбора.
*   `artifacts/best_model_per_target_map.json` — Веса ансамбля для каждого таргета (123 веса).
*   `artifacts/stage2_validation_results.json` — Сравнение моделей, прирост от Stage 1, статистика ансамбля.
*   `models_weight/catboost/stage2_catboost_validation_v1/` — Сохранённая модель CatBoost Stage 2.
*   `models_weight/neural_network/stage2_nn_validation_v1/` — Сохранённая модель NN Stage 2.
*   `models_weight/lightgbm/stage2_lgbm_validation_v1/` — Сохранённая модель LGBM Stage 2.

**Функции внутри:**

*   `model_exists()` — Проверяет наличие `metadata.json` для модели.
*   `load_existing_model()` — Загружает существующую модель без переобучения.
*   `extract_model_oof_from_stacked()` — Извлекает предсказания одной модели из STACKED файла.
*   `calculate_weighted_average_ensemble()` — Создаёт взвешенный ансамбль с глобальными весами.
*   `create_best_model_per_target_ensemble()` — Для каждого таргета выбирает модель с лучшим AUC.
*   `calculate_ensemble_auc()` — Считает macro ROC-AUC для ансамбля.
*   `EnsembleWeightOptimizerPerTarget` (из utils) — Оптимизирует веса через differential evolution для каждого таргета отдельно.

### 5. Инференс и Сабмит (`scripts/04_inference_submission.py`)

Финальная сборка.

*   **Консистентность:** Мета-признаки для теста генерируются точно так же, как для валидации. Никакого рассинхрона.
*   **Фиксация типов данных:** Автоматически приводим `customer_id` к int32, предсказания к float64. Платформы часто отклоняют файлы из-за типов.
*   **Блендинг:** Применяется карта весов, сохранённая на Стадии 2. Сабмит точно соответствует лучшей валидационной стратегии.
*   **Два файла на выходе:** Оригинальный + `_fixed_types`. Для сабмита используем второй.

## 📂 Описание файлов

### `scripts/00_full_data_processing.py`
Гибридный процессор данных на Polars с защитой от утечек, Sparse PCA для пропусков, Target Encoding с регуляризацией, Polynomial и Interaction Features, GroupBy агрегациями, проверкой стабильности признаков и дубликатов, оптимизацией памяти и синхронизацией train/test.

### `scripts/01_generate_folds.py`
Генератор кросс-валидации, создающий воспроизводимые фолды один раз, сохраняющий индексы и метаданные на диск, поддерживающий универсальные пути для работы в любой ОС и стратификацию по сумме таргетов.

### `scripts/02_stage1_proxy_training.py`
Движок обучения базовых моделей (CatBoost, LGBM, NN) для получения OOF предсказаний, с системой чекпоинтов, калибровкой вероятностей изотонической регрессией, генерацией корреляционной матрицы и опциональным adversarial filter.

### `scripts/03_stage2_validation.py`
Скрипт валидации второго уровня, генерирующий мета-признаки на основе корреляций таргетов, отбирающий топ-1500 признаков, оптимизирующий веса ансамбля отдельно для каждого таргета через differential evolution и сравнивающий методы блендинга.

### `scripts/04_inference_submission.py`
Финальный скрипт инференса, загружающий обученные модели всех стадий, генерирующий предсказания на тесте, применяющий оптимизированные веса для блендинга и автоматически исправляющий типы данных в итоговом файле.

### `scripts/06_analyze_models.py`
Инструмент глубокой аналитики, рассчитывающий AUC для каждого таргета отдельно, сравнивающий вклад каждой модели, выявляющий проблемные таргеты, анализирующий корреляцию частоты таргета и AUC, оценивающий влияние NN на ансамбль.

### `scripts/07_pseudo_labeling.py`
Модуль дообучения моделей на тестовых данных, отбирающий наиболее уверенные предсказания (threshold 0.95/0.05), добавляющий их в тренировочную выборку с весом 0.6, переобучающий модели для улучшения обобщающей способности.

### `scripts/check_data_leakage.py`
Сканер безопасности данных, проверяющий признаки на чрезмерную корреляцию с таргетом, сдвиг распределений между train/val, консистентность OOF предсказаний, валидность важности признаков и мета-признаков на утечки.

### `scripts/test_target_gnn.py`
Изолированный тест TabNet + Target GNN с мета-признаками, поддерживающий категориальные эмбеддинги (cat_idxs), Feature Selection, генерацию мета-признаков из OOF и сравнение с baseline CatBoost.

### `scripts/train_tabnet_gnn_single.py`

### `scripts/tune_catboost.py` / `tune_catboost_loop.py`
Оптимизация гиперпараметров CatBoost через умный подбор на основе истории trials, с фиксированными iterations=2000 и depth=7, 12+ настраиваемыми параметрами, сохранением лучших результатов в JSON.

### `scripts/tune_lgbm_single.py` / `tune_lgbm_loop.py`
Оптимизация гиперпараметров LightGBM через умный подбор на основе истории trials, с поддержкой GPU, фиксированными n_estimators=450, 10+ настраиваемыми параметрами, конвертацией numpy типов для JSON.

### `models/catboost_model.py`
Менеджер CatBoost с поддержкой MultiLogloss для 41 таргета, автоматическим определением корня проекта, работой с Polars, сохранением метаданных, агрессивной очисткой памяти и универсальными путями.

### `models/lgbm_model.py`
Менеджер LightGBM с поддержкой GPU, валидацией данных на NaN/Inf, автоматической настройкой параметров в зависимости от редкости класса, сохранением списка валидных таргетов и правильным выравниванием колонок при инференсе.

### `models/nn_model.py`
Менеджер нейросетей на PyTorch с архитектурой MultiLabelNN, взвешенной функцией потерь для дисбаланса классов, единым scaler для всех признаков, сохранением порядка колонок и поддержкой AMP.

### `models/tabnet_gnn_model.py`
Продвинутая модель TabNet + Target GNN с register_buffer для mean_/std_, M_loss из TabNet, отдельным сохранением adjacency матрицы, Early Stopping, AMP, OneCycleLR, извлечением feature importance из attention masks.

### `models/target_gnn.py`
Графовый слой для учёта корреляций между таргетами, с register_buffer для adjacency матрицы, нормализацией строк, residual connections, защитой от NaN/Inf и совместимостью с AMP.

Единая точка входа для данных, кэширующая полные данные в памяти, поддерживающая загрузку фолдов с диска, предоставляющая унифицированный интерфейс для получения train/val/test выборок в формате Polars.

### `utils/ensemble_optimizer.py`
Оптимизатор весов ансамбля с векторизованными вычислениями, алгоритмом differential evolution, кэшированием матриц предсказаний, поддержкой per-target оптимизации и сохранением в JSON.

### `utils/ensemble_optimizer_per_target.py`
Узкоспециализированный оптимизатор, подбирающий уникальные веса ансамбля для каждого из 41 таргета отдельно через differential evolution, позволяющий использовать силу разных моделей там, где они работают лучше.

### `utils/feature_selector.py`
Утилита отбора признаков на основе важности CatBoost, работающая на подвыборке данных для ускорения, возвращающая список топ-N признаков для сокращения размерности на втором этапе.

### `utils/meta_features.py`
Генератор мета-признаков, загружающий корреляционную матрицу таргетов, создающий новые признаки на основе разности предсказаний коррелирующих таргетов, позволяющий моделям второго уровня учитывать взаимосвязи между целями.

### `utils/oof_stacking.py`
Менеджер стекинга, автоматизирующий генерацию OOF предсказаний по фолдам, расчет корреляционной матрицы на их основе, сохранение артефактов для последующего использования на втором этапе.

### `utils/meta_learner.py` / `meta_learner_per_target.py`
Meta-Learner для ансамблирования через Ridge/Logistic/GBM, с оптимизацией весов для каждого таргета отдельно, shrinkage к глобальным весам для защиты от переобучения, проверкой что per-target улучшает global.

Генератор индексов для Cross-Validation с поддержкой Stratified K-Fold для multi-label (стратификация по сумме таргетов), режимом Hold-out для быстрых тестов.

## 💡 Выводы и наблюдения

В процессе работы над задачей с 41 таргетом мы пришли к нескольким важным выводам:

1.  **Единой модели не существует.** CatBoost выигрывает на одних таргетах, Neural Networks — на других, LightGBM обеспечивает стабильность. Per-target оптимизация весов даёт больший прирост, чем усреднение.
2.  **Контекст таргетов важен.** Мета-признаки из корреляций между таргетами дают стабильный прирост. Модель должна "знать" о существовании других таргетов.
3.  **Дисбаланс требует хирургии.** Обычные веса классов работают плохо. Мы внедрили автоматическую адаптацию параметров (min_data_in_leaf, регуляризация) в зависимости от частоты класса.
4.  **Валидация важнее обучения.** Скрипты `check_data_leakage` и `00_full_data_processing` занимают значительную часть кода не просто так. Ошибка в валидации обесценивает любую модель.
5.  **Порядок операций критичен.** Row statistics ДО импутации, Null Flags БЕЗ таргета, Fit ТОЛЬКО на train — эти детали дают суммарно +0.02-0.03 AUC.
6.  **Инженерия > Магия.** 80% успеха — это чистота данных, отсутствие утечек и корректная обработка типов. Сложные архитектуры дают прирост только когда базовый пайплайн отлажен.
